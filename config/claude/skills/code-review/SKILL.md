# Code Review Skill

プロジェクト全体のコードレビューをチーム（複数エージェント）で並行実施し、重要度付きの総合レポートを生成する。

## 実行手順

### Phase 1: プロジェクト分析

1. Explore エージェントでプロジェクトの全体構造を把握する
   - `package.json` / `Cargo.toml` / `go.mod` 等から技術スタックを特定
   - README やドキュメントからプロジェクトの目的を把握
   - ディレクトリ構造からアーキテクチャパターンを特定
   - デプロイ設定（IaC、CI/CD）を確認
2. 上記の結果から、レビュー対象のファイル一覧とレビュー観点を決定する

### Phase 2: レビューチーム編成

TeamCreate でチーム `code-review` を作成し、以下の4名のレビュアーを Task ツールで **並行起動** する。

各レビュアーは `subagent_type: "general-purpose"` で起動し、`run_in_background: true`、`mode: "bypassPermissions"` で実行する。

#### レビュアー構成

| 名前 | 担当観点 |
|------|----------|
| `security-reviewer` | セキュリティ |
| `architecture-reviewer` | モジュール分割・アーキテクチャ |
| `naming-reviewer` | 命名規則・コード品質 |
| `platform-reviewer` | プラットフォーム固有（実行環境に応じて調整） |

### Phase 3: タスク作成・割り振り

TaskCreate で4つのレビュータスクを作成する。各タスクの description には以下を含める：

- レビュー対象ファイルの一覧（全ファイルパス）
- 重点的に確認すべきポイント（具体的に10項目以上）
- 出力形式の指示

### Phase 4: レビュー実行

各レビュアーに以下のプロンプトテンプレートを使って指示する：

```
あなたは「{name}」という名前の{担当観点}レビュー担当エージェントです。

## あなたの担当タスク
TaskListからタスク#{task_id}「{タスク名}」を取得し、その内容に従ってレビューを実施してください。

## 作業手順
1. TaskGet でタスクの詳細を取得
2. TaskUpdate でタスクを in_progress に変更（owner: "{name}"）
3. プロジェクトの全ソースコードを読み、{担当観点}観点でレビュー
4. レビュー結果をチームリードに SendMessage で報告
5. TaskUpdate でタスクを completed に変更

## レビュー対象ファイル
{ファイル一覧}

## 出力形式
日本語で、各指摘事項に重要度（Critical/High/Medium/Low）を付けて報告してください。
ファイルパスと該当行番号も必ず含めてください。
```

### Phase 5: 結果集約・レポート生成

全レビュアーからの報告が揃ったら：

1. 全レビュアーに shutdown_request を送信
2. TeamDelete でチームを削除
3. 以下の形式で **総合レビューレポート** を生成する

## レポート形式

```markdown
# 総合コードレビューレポート

## プロジェクト概要
[プロジェクト名、技術スタック、目的の簡潔な説明]

## レビューチーム構成
[担当者、観点、指摘数のテーブル]

## Critical（即時対応必須）
[重複を統合した上で、各指摘にファイルパス:行番号、指摘者、推奨対応を記載]

## High（早期対応推奨）
[同上]

## Medium（計画的対応）
[テーブル形式で簡潔に]

## Low（必要に応じて対応）
[テーブル形式で簡潔に]

## 良い点
[適切に実装されている箇所のリスト]

## 対応優先度ロードマップ
[即時対応 → 短期 → 中期 → 長期]
```

## レビュー観点の詳細

### 1. セキュリティ（security-reviewer）

必ず以下を確認する：

- 認証・認可（署名検証、トークン管理）
- APIキー・シークレット管理（ハードコード、環境変数）
- 入力バリデーション・サニタイズ
- データベース操作のインジェクション耐性
- 外部API通信のセキュリティ（HTTPS、タイムアウト）
- エラーハンドリングでの情報漏洩
- ログへの機密情報出力（PII、APIキー）
- IAM/権限設定（最小権限原則）
- 依存関係の脆弱性
- OWASP Top 10

### 2. モジュール分割・アーキテクチャ（architecture-reviewer）

必ず以下を確認する：

- レイヤー構造の適切性（循環依存の有無）
- 各モジュールの責務の明確さ（Single Responsibility）
- モジュールの凝集度と結合度
- 型定義・インターフェース設計
- デッドコードの有無
- テスト容易性（DI、モック可能性）
- IaC構成の整合性
- スケーラビリティ・拡張容易性

### 3. 命名規則・コード品質（naming-reviewer）

必ず以下を確認する：

- 命名規則の一貫性（ファイル名、関数名、変数名）
- 関数名・変数名から処理内容が推測できるか
- マジックナンバーの有無
- 型安全性（any型の使用、型ガード）
- コードの可読性（ネスト深さ、関数の長さ）
- コメントの適切さ
- エラーメッセージの品質
- コーディング規約との整合性

### 4. プラットフォーム固有（platform-reviewer）

実行環境に応じて調整する。以下は代表的な例：

**AWS Lambda の場合：**
- コールドスタート最適化
- メモリ・タイムアウト設定の妥当性
- 同時実行制御
- Lambda Context の活用
- 構造化ログ（CloudWatch Logs Insights対応）

**LINE Bot の場合：**
- Webhook応答時間（1秒ルール）
- メッセージAPI制限（Rate limit）
- replyToken の適切な使用
- SDK バージョンの適切さ

**Webアプリの場合：**
- パフォーマンス（バンドルサイズ、レンダリング）
- アクセシビリティ
- SEO
- レスポンシブデザイン

## 重要度の定義

| 重要度 | 定義 | 対応目安 |
|--------|------|----------|
| **Critical** | 本番障害・セキュリティ脆弱性に直結。即時修正必須 | リリース前に必ず修正 |
| **High** | 品質・保守性に大きな影響。早期の対応が必要 | 1-2スプリント以内 |
| **Medium** | 改善により品質が向上するが、即座の影響は限定的 | 計画的に対応 |
| **Low** | 軽微な改善提案。対応は任意 | 余裕があれば対応 |

## ユーザー指定の追加観点

ユーザーが追加のレビュー観点を指定した場合（例：「パフォーマンス」「アクセシビリティ」）、それらを各レビュアーのタスクに組み込むか、追加のレビュアーを作成する。

## 注意事項

- 各レビュアーは対象ファイルを **全て読んでから** レビューを行うこと
- 重複する指摘は統合し、最初の指摘者を記載すること
- 良い点も必ず記載し、バランスの取れたレビューにすること
- レポートのCritical/Highには具体的なファイルパスと行番号を含めること
- Medium/Lowはテーブル形式で簡潔にまとめること
