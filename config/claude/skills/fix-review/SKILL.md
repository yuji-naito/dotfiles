# Fix Review Skill

コードレビューの指摘事項を体系的に修正し、テストで検証しながら安全に適用する。

## 前提条件

- コードレビューレポートが存在すること（`/code-review` スキルの出力、または手動のレビュー結果）
- ユーザーが修正対象の指摘事項を指定すること（重要度、除外項目など）

## 実行手順

### Phase 1: 指摘事項の分析と分類

1. レビューレポートから修正対象の指摘事項を抽出する
2. 以下の基準で修正バッチに分類する：

| バッチ | 内容 | 例 |
|--------|------|----|
| Batch 1 | 即時修正可能な単純修正 | ログからPII除去、エラーメッセージ修正、URL修正 |
| Batch 2 | API/ライブラリ移行 | SDK バージョン移行、非推奨API置換 |
| Batch 3 | 設計変更を伴う修正 | DI導入、モジュール分割、新ファイル作成 |
| Batch 4 | インフラ・設定変更 | IaC修正、CI/CD設定、環境変数 |

3. 各バッチの修正内容と影響範囲をユーザーに提示する

### Phase 2: 事前準備

1. 修正対象の全ファイルを読み込む
2. 現在のテスト状況を確認する（`npm test` または該当するテストコマンド）
3. テストが全て通ることを確認してから修正を開始する

```bash
# テストコマンドはプロジェクトに応じて調整
npm test
```

### Phase 3: バッチごとの修正実行

各バッチについて以下のサイクルを実行する：

#### 3.1 修正適用
- 関連ファイルを読み込み、指摘事項に対応するコード修正を適用する
- 修正は Edit ツールを使用し、最小限の変更に留める
- 1つのバッチ内で関連する修正をまとめて適用する

#### 3.2 型チェック
```bash
npx tsc --noEmit
```
- 型エラーがある場合は即座に修正する

#### 3.3 テスト実行
```bash
npm test
```
- テスト失敗がある場合：
  - **修正内容に起因するテスト期待値の変更** → テストを更新する
  - **修正によるロジック破壊** → 修正内容を見直す
- 全テストが通るまでこのバッチを完了しない

#### 3.4 バッチ完了報告
- 修正したファイルの一覧
- 変更の概要
- テスト結果

### Phase 4: 最終検証

全バッチ完了後：

1. 全テスト実行で合格を確認
2. 型チェック通過を確認
3. Lint チェック（設定がある場合）
4. 修正サマリーを生成

## 修正サマリー形式

```markdown
# レビュー指摘修正サマリー

## 修正対象
- Critical: X件
- High: X件
- 除外: [除外した指摘とその理由]

## バッチ別修正内容

### Batch 1: 即時修正
| ファイル | 修正内容 | 対応する指摘 |
|----------|----------|-------------|
| ... | ... | ... |

### Batch 2: API/ライブラリ移行
| ファイル | 修正内容 | 対応する指摘 |
|----------|----------|-------------|
| ... | ... | ... |

### Batch 3: 設計変更
| ファイル | 修正内容 | 対応する指摘 |
|----------|----------|-------------|
| ... | ... | ... |

### Batch 4: インフラ・設定変更
| ファイル | 修正内容 | 対応する指摘 |
|----------|----------|-------------|
| ... | ... | ... |

## テスト結果
- 全テスト: X件 通過
- 修正に伴うテスト更新: X件

## 未対応の指摘
| 指摘 | 理由 |
|------|------|
| ... | ... |
```

## 修正時の原則

### やるべきこと
- 修正は最小限に留める（指摘された箇所のみ変更）
- 各バッチ後に必ずテストを実行する
- テスト期待値の変更が必要な場合は、変更理由を明確にする
- 外部APIのコード値等は公式ドキュメントで検証してから修正する
- DI導入時はオプショナル引数パターン（`param?: Type`、デフォルトで `new Type()`）を使用する

### やってはいけないこと
- 指摘されていない箇所の「ついで修正」
- テストが通らない状態で次のバッチに進む
- テストを削除して通す
- 型エラーを `as any` で回避する
- 修正の影響範囲を確認せずに変更する

## ユーザー指定の除外項目

ユーザーが特定の指摘を除外した場合（例：「Webhook応答タイムアウト以外を修正して」）、その指摘は修正サマリーの「未対応の指摘」セクションに理由とともに記載する。

## 注意事項

- プロジェクトの CLAUDE.md が存在する場合は、そのコーディング規約に従う
- 修正によって新たな問題が生じた場合は、修正を中止してユーザーに報告する
- 大規模な設計変更（ファイル分割、アーキテクチャ変更など）は事前にユーザーの承認を得る
